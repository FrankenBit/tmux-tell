### Added

- Documentation now reflects that the **Codex adapter (`tmux-tell-codex`) is a full paste-and-Enter delivery peer**, not just a hook-context recipient: the README names both the Claude and Codex adapters (and `--adapter=codex` at install time), and the asciinema-capture F6 finding gets a Codex sibling note pointing at the per-adapter `PaneProfile`. The paste-delivery capability itself shipped incrementally (#322 sentinel + cursor model, #360 paste-and-enter default, #401 two-Enter resubmit, #533 footer normalization); this closes the docs gap that completed the #278 Codex-paste arc.
- Bus-mediated `clear` now requires a `--for-task` / `for_task` target-task label (e.g. `tmux-tell#286`) and synthesises an atomic `/clear` + `/rename "<Chamber> <task>"` pair ŌĆö clearing the recipient's session and then relabelling the fresh session to its dispatch-time task identity (#286). This keeps the chamber's persistent name free so a later `claude --resume <Chamber>` resolves unambiguously instead of colliding with the orphaned same-named session a bare `/clear` leaves behind. The label is constrained (single line, starts alphanumeric, Ōēż80 chars of `[A-Za-z0-9 #/._-]`) so nothing a caller types can smuggle a second pasted command; `for_task` on any command other than `clear` is rejected fail-loud, and `clear` without it is rejected (forward-only ŌĆö the bosun/qmŌåÆpilot edges are updated in lockstep, no deprecation cycle). The post-`/compact` settle-gate now also covers `/clear`, so the follow-up `/rename` lands in the settled new session rather than being swallowed mid-reset.
- Remote MCP mode (#310): set `$TMUX_TELL_REMOTE_HOST=[user@]host[:port]` (default port 7777) and the MCP server forwards every tool call back to the originating bus's `tmux-tell-claude` over a reverse-SSH tunnel instead of opening a local store ŌĆö closing the return path for a Claude session you've SSH'd into from your local tmux. It's an explicit opt-in deviation from the MCP-and-DB-share-a-machine default: without the env var the MCP behaves as a local standalone, and remote mode is never inferred. One universal forwarder mirrors the full tool surface and re-runs the actual handler on the originating host (via the hidden `__remote-mcp-recv` receiver), so results are identical to a local call with no per-tool translation ŌĆö pane-introspection (`agents`/`whoami`/`agent_state`) naturally reports the originating host's tmux, which is what a remote participant wants. Identity is the remote session's tmux session name (`#S`, which Claude/codex set to the chamber name) or an explicit `$TMUX_AGENT_NAME`; with neither resolvable the server fails loud at startup, and the resolved identity must be a single shell-safe token (it crosses the originating host's login shell via ssh, so whitespace or shell metacharacters are rejected up front rather than truncating or injecting at forward time). SSH failures are fail-loud per tool call, never silent. See `docs/reference.md` ┬¦ "Remote MCP via reverse-SSH". Cross-host bus federation remains out of scope.
- The agent manual's self-`/compact` section now spells out *why* the atomic `sleep` + `resume_with` verb is preferred over filing a hand-off and then hand-typing `/compact`: the two-step path has a stall-window (hand-off staged, reset never fired ŌĆö the "stalled at the execution seam" pattern), and the atomic verb collapses both into one call so nothing is left half-done. This is the discipline framing #615 asked for; the underlying atomic mechanism (a `/compact` row + a deferred `resume_with` row inserted together via `InsertMessagePair`) already shipped and is hardened by the #622 stability-gate.
- The mailman now **auto-resumes a chamber after a transient rate-limit** (#618): when the #448 self-observe probe sees `StateRateLimited`, it waits out the cooldown (banner `Retry-After` hint or the #613 exponential backoff, plus #543 jitter) then pastes `continue` to resume the interrupted turn ŌĆö the prompt the operator typed by hand before. It rides the existing self-observe cadence (like #621's metabolism auto-clear), so it fires regardless of whether a bus message is queued; the probe loop is the verify-and-retry, ending the episode once the chamber leaves rate-limited. Bounded by `--rate-limit-resume-max-attempts` (default 5) before giving up and leaving the chamber for the operator. Only the #504 transient throttle is resumed ŌĆö `StateUsageLimited` (#540 hard quota park) is left alone. Default-on with no don't-fight-upstream guard (no adapter auto-resumes natively today); `--rate-limit-resume-disabled` is the escape hatch. New counter `tmux_tell_rate_limit_resume_total{outcome,agent,provider}` (`attempt` / `recovered` / `gave_up`). Detection rides the existing `RateLimitPattern` substrate; the empirical Claude/Codex pane patterns remain a separate forcing-function-gated capture (#504/#540).
- First-class **metabolism** ŌĆö a chamber's self-reported, intentional context-throughput state, the layer-3 axis that complements the auto-observed `agent_state` probe and the operator-flag `attention_state`. Three states the pane-observer cannot infer: `warming` (just resumed, not yet at full throughput), `saturating` (context-load approaching the `/compact`-need), `compact-pending` (intent-to-`/compact` stated but not yet fired). A chamber self-reports via `tmux-tell.set_metabolism` (MCP) or `tmux-tell-claude set-metabolism <value>` / `--clear` (CLI) ŌĆö **self-only**: the call resolves the caller's own identity and exposes no target, so a chamber can only set its own metabolism (a third-party write would clobber the target's real signal). The self-report is advisory (it never gates delivery) and surfaces alongside the observed state in `agent_state`, plus a `METABOLISM` column in `agents`. `compact-pending` auto-clears once the mailman observes the chamber actually `at-rest-in-compaction` (observed-truth supersedes the redundant self-report). ### Fixed

- `track --watch` now exits cleanly (status 0) when its `--timeout` elapses, instead of occasionally returning `exitInternal` (70) with a `context deadline exceeded` error (#281). The watch loop's store read could observe the loop's own timed-out context as a `DeadlineExceeded` failure before the select reached its `ctx.Done()` branch; that self-cancellation is now recognized as the clean exit path. This was the production race behind the `-race`-only `TestRunTrackWatch_TimeoutExits` flake; the test is also re-anchored on `context.WithTimeout`'s deterministic deadline (a floor + a generous jitter-immune ceiling) instead of a tight wall-clock bound that host load could blow past.
