### Changed

- Release-cut machinery now consumes [`frankenbit/release-toolkit`](https://git.frankenbit.de/frankenbit/release-toolkit) (`reusable-release.yml` + `reusable-manifest-check.yml` at `@v0.18.0`) instead of the hand-rolled in-repo workflows (#617). Cuts are now push-triggered: a release-relevant push to `main` opens or updates a rolling release-prep PR, and merging it cuts the release. `publish_mode: draft` preserves the existing operator-Publish gate -- the cut creates a draft release; clicking Publish fires `release: published` and chains `deploy.yml` onto alcatraz-host unchanged. `workflow_dispatch` is retained as an emergency-cut fallback. The `changelog.d/<issue>.<type>.md` fragment format, the pre-1.0 minor-carries-breaking cadence, and the README `--version`-pin sync (now a release-toolkit post_bump_hook) all carry over intact.
- Added `workflow_dispatch:` trigger to `.forgejo/workflows/test.yml` so an operator can manually re-run the lint+build+test job from the Forgejo Actions UI. Catches the class of failure where the automatic `pull_request` or `push` trigger doesn't fire — most notably when release-toolkit's release-bot pushes a `chore(release): prepare vX.Y.Z` commit to `release-prep/rolling` using its system token, which doesn't trigger downstream workflows (anti-recursion safeguard same as GitHub Actions' GITHUB_TOKEN rule). v0.23.0's rolling PR (#630) hit exactly this on the first toolkit-driven cut.

### Fixed

- Capped `--workers=4` on all three remaining `gremlins-*` jobs in `mutation.yml` (`render`, `store`, `tmuxio`). The `gremlins-tmuxio` job peaked at 9.7 GB on 2026-06-28 ~05:00 UTC and triggered a systemd-oomd cascade that crashed alcatraz (3.5h offline). Same defense-in-depth shape that `gremlins-cli` already had per the prior characterisation run. Render + store haven't OOM'd historically but the precautionary cap is free.
- Extended the `check-changelog-placement` skip pattern in `.forgejo/workflows/test.yml` to cover `release-prep/rolling` — release-toolkit's auto-opened rolling PR branch name. The previous regex only matched `i/v[0-9]*` (manual-cut convention) and `release-prep/v[0-9]*` (v0.22.0-era release.yml bot branch); release-toolkit (adopted in v0.23.0 via #617) uses the literal branch name `release-prep/rolling`, which wasn't matched, so the placement check would fail on every rolling PR claiming the legitimate cut additions to `## [X.Y.Z]` were rebase-past-cut injections. Caught empirically on the first toolkit-driven cut (#630).
