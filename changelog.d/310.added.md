Remote MCP mode (#310): set `$TMUX_TELL_REMOTE_HOST=[user@]host[:port]` (default port 7777) and the MCP server forwards every tool call back to the originating bus's `tmux-tell-claude` over a reverse-SSH tunnel instead of opening a local store — closing the return path for a Claude session you've SSH'd into from your local tmux. It's an explicit opt-in deviation from the MCP-and-DB-share-a-machine default: without the env var the MCP behaves as a local standalone, and remote mode is never inferred. One universal forwarder mirrors the full tool surface and re-runs the actual handler on the originating host (via the hidden `__remote-mcp-recv` receiver), so results are identical to a local call with no per-tool translation — pane-introspection (`agents`/`whoami`/`agent_state`) naturally reports the originating host's tmux, which is what a remote participant wants. Identity is the remote session's tmux session name (`#S`, which Claude/codex set to the chamber name) or an explicit `$TMUX_AGENT_NAME`; with neither resolvable the server fails loud at startup, and the resolved identity must be a single shell-safe token (it crosses the originating host's login shell via ssh, so whitespace or shell metacharacters are rejected up front rather than truncating or injecting at forward time). SSH failures are fail-loud per tool call, never silent. See `docs/reference.md` § "Remote MCP via reverse-SSH". Cross-host bus federation remains out of scope.
