`track --watch` now exits cleanly (status 0) when its `--timeout` elapses, instead of occasionally returning `exitInternal` (70) with a `context deadline exceeded` error (#281). The watch loop's store read could observe the loop's own timed-out context as a `DeadlineExceeded` failure before the select reached its `ctx.Done()` branch; that self-cancellation is now recognized as the clean exit path. This was the production race behind the `-race`-only `TestRunTrackWatch_TimeoutExits` flake; the test is also re-anchored on `context.WithTimeout`'s deterministic deadline (a floor + a generous jitter-immune ceiling) instead of a tight wall-clock bound that host load could blow past.
