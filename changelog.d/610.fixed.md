Bus messages to codex chambers no longer stack as unsubmitted text in the composer when codex is under load (#610). The mailman delivers via `paste-and-enter`; under high context/output load codex is still ingesting and collapsing a large paste (`[Pasted Content …]`) when the submit-Enter fires, so the per-delivery verify budget exhausts with the paste still unsubmitted. The **next** delivery then pasted on top of it — the #616 pre-paste race check is cursor-anchored and can't see a stuck collapsed paste (codex parks the cursor on an empty sub-line so the input row reads "cleared"), so it was scoped to operator drafts and left the collapse marker to the #401 resubmit machinery, which under load runs out of budget. Messages accumulated into one corrupted ~15-line composer that the operator had to submit by hand. The pre-paste check now also detects a stuck collapsed paste (the adapter's `PasteCollapseMarker` in the bottom-most input block): rather than paste onto it, the mailman fires one resubmit Enter to drain it — extending the #401 resubmit across mailman cycles, not just within one delivery's budget — and reverts the current message to queued for a later cycle. Stacking/corruption becomes clean serialization, and the stuck paste drains without the operator as the backstop. Codex-specific by config (Claude has no collapse marker, so it is unaffected). Reduces the upstream cause of #609 (a stuck composer keeps the prompt row non-empty, so the classifier reads `unknown` and the mailman defers all delivery — fewer stuck composers, fewer dark chambers). The deeper load-adaptive submit-reliability work (stability-gated resubmit, load-scaled budget) is tracked separately as a follow-up.
